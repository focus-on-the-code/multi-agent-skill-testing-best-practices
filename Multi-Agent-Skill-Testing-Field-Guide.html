00 — Why this exists

A real skill test became a reusable practice

This guide grew from testing Agentic Project Kickoff. The goal was not a theatrical agent demo; it was to learn whether the skill could guide a genuinely fresh agent while keeping a human project owner in control.

The skill under test

Agentic Project Kickoff

A reusable facilitation workflow for beginning software, app, automation, AI-workflow, and agentic-coding projects thoughtfully.

  • Runs adaptive project discovery.
  • Creates project-specific planning and workspace documents from templates.
  • Preserves decisions, assumptions, and open questions.
  • Builds understanding across phases without rushing into implementation.
1

Why sub-agents?

An agent that already knows the expected behavior can unconsciously reproduce the answer key. A clean child agent showed whether the skill itself—not prior context—caused the right behavior.

2

How we ran it

Agent 1 acted as the project owner and conductor for a children’s physics website. It supplied simple answers under an at-most-five-question test control while a fresh child used the skill.

3

How we checked it

Agent 1 reported the behavioral run. Agent 2 independently reviewed the skill and evidence, revealing what the final artifacts proved—and what they could not prove about timing.

4

What changed

The skill gained narrower approval semantics and the heading “Current Discovery Summary.” The test method gained a live parallel observer, gate snapshots, timestamps, and source hashes.

Why it matters

We turned one pass-with-caveats into two kinds of improvement: a clearer skill for human users and a more trustworthy way to test skills. We rejected a proposed phase-by-phase validation ritual because test rigor should strengthen the evaluation—not add paperwork to normal human use.

01 — Foundation

Six rules that keep a test honest

The test should reveal whether the skill generalizes in a fresh context, not whether another agent can reconstruct an expected result from leaked instructions.

1

Separate roles

The project owner and the system under test must be different agents.

2

Protect context

Give the child only the skill, a natural request, a disposable path, and safety bounds.

3

Observe before judging

Let behavior emerge before showing the child checklists, defects, or conclusions.

4

Split evidence types

Structure and hashes are deterministic; activation and gate discipline are behavioral.

5

Preserve raw proof

Keep the transcript, gate trees, timestamps, artifacts, and source comparisons.

6

Label test controls

A five-question cap can bound a test without becoming a requirement of the skill.

02 — Architecture

One test, four active roles

Select a role to see its mandate. Agent 2 runs in parallel so staged timing becomes independently observed evidence.

Primary orchestrator

Defines the contract, records source hashes, starts the live reviewer, verifies both reports, and cleans up the disposable project.

04 — Key improvement

The reviewer watches live

A post-test reviewer can inspect the destination. Only a live observer can independently verify when files appeared.

Neutral gate channel

  1. Agent 1 sends the exact child response and announces the gate.
  2. Agent 2 inspects the filesystem before approval.
  3. Agent 2 records the initial tree, hashes, and timestamp.
  4. Agent 1 sends the exact narrow approval.
  5. Agent 2 records when expansion first becomes visible.

Important: the observer sees events and evidence; the child never sees the answer key.

not narrative.
Baseline

Agent 2 records the source hash manifest and empty or known disposable-project state before the child begins.

05 — Runbook

Thirteen controlled moves

Expand any step for the operating rule. The sequence keeps context clean while preserving enough evidence for a real audit.

1. Define the contract

Set scope, prohibitions, paths, models, question cap, reports, and cleanup policy.

2. Run deterministic prechecks

Inspect structure, syntax, references, permissions, and pre-test hashes.

3. Launch Agent 1

Make it the conductor and project owner—not the system under test.

4. Launch Agent 2

Start the live observer before the child; record baselines and wait for gate events.

5. Launch the child

Provide only the skill, disposable path, natural request, and safety boundaries.

6. Conduct discovery

Agent 1 answers naturally from an answer bank and labels any question cap as a test control.

7. Capture the first gate

Both agents record the tree; Agent 2 hashes initial artifacts before continuation.

8. Approve narrowly

Authorize only the next stage. Expansion approval does not approve every proposed decision.

9. Capture completion

Agent 2 records first expansion, final tree, hashes, and prohibited-output checks.

10. Agent 1 reports

The conductor compares observed behavior with the skill and labels inference versus proof.

11. Agent 2 finalizes

The observer checks Agent 1’s report against its live log and raw artifacts.

12. Primary audit

The orchestrator resolves disagreements, checks source integrity, and assigns statuses.

13. Clean up

Remove only the disposable project after evidence and reports are retained.

06 — Case study

What two agents taught us

The first report demonstrated strong staged behavior. The second showed why independent evidence and precise scoring still matter.

Agent 1 · Behavioral test

Pass, with minor caveats

  • Fresh child followed staged kickoff.
  • Discovery stopped after two useful questions.
  • Only the initial artifact set existed before approval.
  • Expansion remained planning-only.
  • No project README or local playbook appeared.
Agent 2 · Independent review

Conditional pass

  • Final workspace was structurally strong.
  • A final snapshot could not prove earlier timing.
  • The five-question cap was a test rule, not a skill rule.
  • A broad approval created decision-status ambiguity.
  • Hash evidence should replace Git when Git is unavailable.
Resulting improvements

Keep the adaptive skill. Strengthen the test.

The skill gained clearer approval granularity and a more accurate “Current Discovery Summary.” The testing method gained a live Agent 2 observer, immutable gate evidence, source manifests, and explicit separation between test controls and skill requirements.

07 — Anti-patterns

Failure modes worth catching

Search the list by symptom, role, or evidence type.

Conductor becomes the SUT

Fix: Agent 1 launches a separate fresh child.

Answer key leaks into context

Fix: initially share only the skill and natural request.

Final tree “proves” timing

Fix: Agent 2 captures each gate live before approval.

Reports replace evidence

Fix: retain transcripts, manifests, timestamps, and hashes.

Test cap becomes a skill rule

Fix: derive scoring criteria from the skill itself.

Approval grows in scope

Fix: authorize the next stage narrowly; keep decisions proposed.

No Git repository

Fix: use before-and-after source hash manifests.

Agent stalls before execution

Fix: check live state and files, bound no-progress time, then restart or report infrastructure failure.

Testing burdens normal users

Fix: keep audit rigor in the test harness, not everyday skill use.

Reviewer arrives too late

Fix: launch Agent 2 before the child and maintain a neutral event channel.

09 — Reuse

Prompt scaffolds

These outlines establish roles and evidence boundaries without feeding expected behavior to the child.

Agent 1 · Conductor

Act as the independent test conductor and simulated project owner.

Skill: [PATH]
Disposable project: [PATH]
Report: [PATH]

Spawn a fresh child using [MODEL / REASONING]. Give it only the skill, project path, and natural user request.

Answer one question at a time from the answer bank. Enforce [CAP] only as a test constraint. Notify the live reviewer at every response and gate. Do not continue until its snapshot is complete.

Agent 2 · Live reviewer

Act as a live independent observer and evidence reviewer.

Read the skill and record source/project baselines before the child starts. Watch through the neutral event channel.

At each gate, record exact messages, tree, hashes, and timestamps before Agent 1 continues. After the run, compare Agent 1's report with your live log and raw artifacts. Separate skill defects from test limitations.
